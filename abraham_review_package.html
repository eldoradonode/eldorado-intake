01

System Overview

Eldorado Node Hiring Intelligence is a production-grade AI evaluation layer for corporate hiring teams. It accepts CV files, recorded interviews, and recruiter intelligence notes, then produces a structured candidate report with scores, verdicts, compliance flags, interview questions, and an offer or rejection email, all within 2 to 5 minutes of submission.

The system is not a generic AI wrapper. It is a deterministic workflow orchestration layer where each decision step is a discrete, auditable function. Every scoring dimension, every compliance check, every email dispatch is a named node with defined inputs and outputs. This makes the decision trail reconstructable at any point.

Core design principle: No single AI inference produces a final decision. Scoring passes through a structured validation node that enforces score bounds, checks for mandatory fields, and flags any response that does not conform to the expected schema before the verdict is written to the audit log.

The system runs as a multi-tenant SaaS. Each client organisation has its own API key, its own Google Drive folder for candidate files, and its own branded email output. Candidate data never touches a shared pool. The architecture enforces this at the storage routing level, not at the application layer.

02

Architecture & Module Map

The system is composed of seven functional modules, each built as a self-contained node chain within the orchestration layer.

Intake & Auth
Receives multipart form submission. Extracts API key from headers. Validates against client registry. Checks quota. Rejects with structured error if auth fails or quota is exhausted. No data is written to storage before auth passes.
Document Processing
CV is uploaded to the client's own Google Drive folder. Interview recording is transcribed using Groq Whisper. CV text is extracted and parsed into structured schema by Gemini. Files go to client Drive, never to a shared pool.
Intelligence Gathering
Tavily searches LinkedIn and web footprint using candidate name and provided LinkedIn URL. Results are passed as context into the scoring prompt. Professional footprint accuracy improves significantly when LinkedIn URL is provided at intake.
AI Scoring Engine
Gemini evaluates six scoring dimensions against role-specific benchmark templates. Calibration mode adjusts weighting per client tier and role type (velocity for Growth/Sales, structural for Enterprise, strategic for Enterprise Custom/senior roles). Output is a raw JSON score object.
Validation & Audit
A deterministic code node validates score bounds (0–100), enforces all required fields, and computes the final weighted score. Verdict thresholds: HIRE ≥ 75, REVIEW 55–74, PASS below 55. Results are written to the audit log with full scoring breakdown, compliance flags, and timestamp. This node cannot be skipped.
Output Generation
Generates HR report, offer or rejection email, 30/60/90 day onboarding plan, follow-up sequence, and interview scorecard PDF (Growth+ only). All email output carries client branding. No Eldorado Node branding is visible to candidates.
Billing & Lifecycle
Stripe and Paystack webhooks trigger new client creation, quota reset, and top-up. Drip email sequence fires automatically on account creation. Quarterly performance reports auto-send to Enterprise clients. Trial to paid upgrade handled entirely by payment webhook.

Enforcement & Compliance Gates

The following gates are deterministic code nodes, not AI inferences. Each runs regardless of AI output quality.

Gate What It Enforces Failure Behaviour
Auth + Quota Gate Valid API key, active account status, quota units remaining Returns structured 401/403/429 error. No data written.
Score Validation Gate All six dimensions present, values 0–100, weighted sum within tolerance Flags submission as REVIEW, logs schema_error flag, continues to report.
Compliance Rules Engine Red flag pattern matching on transcript and CV: employment gap thresholds, consistency triggers, language mismatch detection Sets complianceStatus to FLAG or REVIEW. Adds flag reasons to audit row.
Tier Feature Gate ATS push, scorecard PDF, compliance export, calibration mode , all check client tier before executing Branch skips silently. Client receives only what their tier permits.
Audit Log Write Every evaluation writes a full row before any output is sent to the HR manager If write fails, execution halts. No report is sent without an audit record.
Data Sovereignty Router CV and recording always upload to the client's own Drive folder ID, never a shared pool If folderId is missing, execution halts at upload step with 500 error.
06a

Cryptographic Execution Boundary

The boundary between the payment layer and account creation is enforced cryptographically. No client record is written, no API key is generated, and no activation email is sent until the inbound webhook payload has been verified against a shared secret using HMAC-SHA256. If verification fails, execution stops at that node. There is no fallback path that permits account creation from an unverified event.

This gate exists because the billing webhook is a public endpoint. Any actor who discovers the URL could attempt to POST a fabricated payment event and receive a live API key. The cryptographic gate makes that structurally impossible regardless of what the payload contains.

Fail-closed by design. The verification node throws a hard error on signature mismatch. n8n does not catch and continue, it stops that execution branch entirely. The webhook responds with a 200 to the provider (to prevent replay retries) but nothing downstream executes. This is the same pattern Stripe recommends for production webhook handling.
Boundary Mechanism Failure Behaviour What It Prevents
Stripe inbound HMAC-SHA256 of timestamp + raw body against STRIPE_WEBHOOK_SECRET Hard throw. Execution halts. No downstream nodes run. Fabricated payment events, replayed events, tampered payload bodies
Paystack inbound HMAC-SHA256 of raw body against PAYSTACK_SECRET_KEY Hard throw. Execution halts. No downstream nodes run. Forged charge.success events, spoofed payment confirmations
Raw body preservation Webhook node configured with rawBody: true to prevent body parsing from altering the signature input If raw body unavailable, verification falls back to serialised JSON Signature drift caused by JSON normalisation differences
Provider identification Switch node routes Stripe and Paystack to separate verification paths before any shared logic runs Unknown provider causes execution halt at switch node Cross-provider payload injection, mixed event handling

This pattern is directly analogous at the agent execution layer: the action does not execute unless the authorisation token is cryptographically valid at the moment of execution, not at the policy layer. In this system, the account creation action does not execute unless the payment event is cryptographically valid at the moment it arrives. The trust is not assumed from the source IP or the URL path. It is verified from the payload signature on every single request.

07

Known Fragility Points

The following are acknowledged fragility points assessed with honest severity. This is not a list of bugs. It is a list of structural conditions that could cause degraded behaviour under specific circumstances.

Medium
Google Drive setup requires manual client action
Each new client must create a Drive folder, share it with the upload service account, and provide the folder ID at checkout. If they misconfigure this (wrong permissions, wrong ID), the file upload fails. There is a 60-second in-form guide but no automated validation before payment. Mitigation: post-payment Drive validation webhook is on the hardening list.
Medium
AI scoring relies on a single inference call
The scoring engine makes one Gemini call per evaluation. There is no ensemble voting or second-pass verification. A hallucinated score that falls within valid bounds (0–100) will pass the validation gate. The narrative verdict provides a soft audit trail but does not catch numerical drift. Planned mitigation: confidence score threshold check and optional re-scoring flag for human review.
Medium
Industry calibration is logic-based, not ML-trained
The scoring weight adjustments for VELOCITY, STRUCTURAL, and STRATEGIC calibration modes are hand-coded thresholds, not derived from historical outcome data. They are directionally correct but have not been validated against real hiring outcomes. For regulated environments, this is a documentation gap until outcome correlation data is available.
Low
Quarterly performance report is newly built, not battle-tested
The quarterly auto-email to Enterprise clients runs on a scheduled cron. It has been built and connected but has not executed through a full quarterly cycle in production. The data aggregation logic is correct in testing but edge cases (clients with zero evaluations in a quarter, timezone boundary issues in date filtering) have not been stress-tested.
Low
No rate limiting on the intake webhook beyond quota
The auth gate checks quota units but does not enforce a per-minute submission rate. A client could submit 50 evaluations simultaneously, which would exhaust quota instantly and create a spike in concurrent AI inference calls. For current usage volumes this is not a risk, but it becomes one above 20 concurrent clients.
Low
Client portal authentication is session-only, no token refresh
Portal sessions are stored in sessionStorage. There is no persistent token or refresh mechanism. A browser restart or tab close ends the session and requires re-authentication with the API key. For enterprise HR teams using shared devices, this could be a friction point.
08

Hardening Priorities

The following are the items I am prioritising for hardening before scaling to enterprise clients, in order of impact.

P1
Drive folder validation at onboarding
Add an automated check immediately after new client creation that pings the provided folder ID and confirms write access before the activation email is sent. Currently this is discovered only on first submission.
P2
Scoring confidence threshold + re-score flag
Add a confidence score output to the AI scoring prompt. If confidence is below a defined threshold, auto-flag the submission for human review before the HR report is sent. This is the primary gap between current behaviour and what regulated environments require.
P3
Intake webhook rate limiting
Add a per-client per-minute submission cap (configurable by tier) enforced at the auth gate. Prevents abuse and protects AI inference costs at scale.
P4
Quarterly report edge case hardening
Test the scheduled quarterly report against zero-activity clients, timezone boundary dates, and clients created mid-quarter before the first live cycle runs in July 2026.
P5
Portal session persistence for enterprise teams
Add a secure persistent token option (remember this device) for enterprise clients. Low urgency until enterprise accounts are active.
On the documentation gap: For regulated enterprise environments, the primary gap is not the existence of an audit trail (which the system has) but its accessibility. The compliance export webhook returns a raw CSV. A human-readable audit report per candidate, exportable as PDF with evaluator version and score justification attached, would close the legal defensibility gap for most regulated jurisdictions. That is the next structural build after the P1 and P2 items above.
